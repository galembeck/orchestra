# Administrators — Plataforma Page Design

**Date:** 2026-05-16
**Status:** Approved

## Overview

Add an "Administradores" section to the admin dashboard sidebar with a "Plataforma" submenu page that lists all internal admin users and allows authorised admins to update each user's role and per-user permission overrides via a slide-over sheet.

---

## 1. Database Schema

### New table: `admin_roles`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | auto-generated |
| `key` | varchar(50) unique | machine identifier: `owner`, `super_admin`, `developer`, `manager`, `moderator`, `analyst`, `billing`, `people`, `viewer` |
| `name` | varchar(100) | Display name |
| `description` | text | Nullable |
| `permissions` | jsonb | Default permission matrix for this role |
| `createdAt` | timestamp | defaultNow() |
| `updatedAt` | timestamp | set on each write |

### Changes to `admin_users`

Two new columns added via Drizzle migration:

- `roleId` — uuid, nullable FK → `admin_roles.id`
- `permissionOverrides` — jsonb, nullable — stores only the user-level deviations from the role defaults. `null` means the user inherits the role's matrix exactly.

### Permission matrix shape

Used in both `admin_roles.permissions` and `admin_users.permissionOverrides`:

```ts
type PermissionMatrix = {
  companies:  { view: boolean; approve: boolean; reject: boolean };
  adminUsers: { view: boolean; create: boolean; updateRole: boolean; updatePermissions: boolean; deactivate: boolean };
  settings:   { view: boolean; edit: boolean };
};
```

Effective permissions for a user = role defaults deeply merged with `permissionOverrides` (user values win on conflict).

## 2. API Endpoints (`apps/api-admin`)

All routes require the `authenticate` preHandler (existing middleware).

| Method | Path | Description |
|---|---|---|
| `GET` | `/admin-users` | List all admin users with role and permission overrides |
| `PATCH` | `/admin-users/:id/role` | Assign a role to an admin user |
| `PATCH` | `/admin-users/:id/permissions` | Set per-user permission overrides |
| `GET` | `/admin-roles` | List all roles with their full permission matrix |

### `GET /admin-users` response item shape

```ts
{
  id: string;
  name: string;
  email: string;
  registration: string;
  avatarUrl: string | null;
  isActive: boolean;
  createdAt: string; // ISO date
  role: { id: string; key: string; name: string } | null;
  permissionOverrides: Partial<PermissionMatrix> | null;
}
```

### `PATCH /admin-users/:id/role` body

```ts
{ roleId: string }
```

### `PATCH /admin-users/:id/permissions` body

```ts
Partial<PermissionMatrix> | null
```

`null` clears all overrides, reverting the user to pure role defaults.

### `GET /admin-roles` response item shape

```ts
{
  id: string;
  key: string;
  name: string;
  description: string | null;
  permissions: PermissionMatrix;
}
```

## 4. Frontend (`apps/admin`)

### Navigation changes

**`src/constants/_app/panel/panel-sidebar.ts`**

The "Administradores" entry becomes collapsible — remove its top-level `url` and add an `items` array:

```ts
{
  title: "Administradores",
  icon: ShieldUser,
  items: [
    { title: "Plataforma", url: "/panel/administrators/platform" }
  ]
}
```

**`src/constants/_app/panel/panel-navigation.ts`**

Update the administrators search entry URL to `/panel/administrators/platform`.

### New route

`src/pages/_app/panel/administrators/platform/index.tsx`
→ resolves to `/panel/administrators/platform`

Route token: `createFileRoute("/_app/panel/administrators/platform/")`.

### Component tree

```
administrators/platform/
  index.tsx
  ~components/
    -administrators-table.tsx
    -administrator-row-menu.tsx
    -edit-permissions-sheet.tsx
```

#### `index.tsx`
- Fetches `adminUsersQueryOptions` and `adminRolesQueryOptions` in parallel.
- Passes data down to `AdministratorsTable`.
- Wires `updateAdminUserRole` and `updateAdminUserPermissions` mutations with `toast` feedback and query invalidation.

#### `-administrators-table.tsx`
Table columns: **Name**, **E-mail**, **Matrícula**, **Cargo** (role name badge), **Status** (active/inactive badge), **Criado em**, **Actions** (kebab menu).

Empty state shown when no users are returned.

#### `-administrator-row-menu.tsx`
Kebab menu per row. Action: "Editar permissões" — opens the edit sheet for that user.

#### `-edit-permissions-sheet.tsx`
Slide-over `Sheet` component. Contains:

1. **Role selector** — dropdown populated from `GET /admin-roles`. Shows current role.
2. **Permission matrix** — grouped by domain (`Empresas`, `Administradores`, `Configurações`). Each action is a toggle switch.
   - Toggles initialised from resolved permissions (role defaults merged with user overrides).
   - Visually marks overrides (e.g. italic label or a small indicator dot) so it's clear which values deviate from the role default.
3. **"Salvar" button** — fires role mutation (if role changed) and permissions mutation (if overrides changed). Closes sheet on success.
4. **"Limpar substituições" button** — sends `null` to `PATCH /admin-users/:id/permissions` to revert to pure role defaults.

- Creating new admin users from the frontend (endpoint already exists, no UI yet).
- Role management page (CRUD for `admin_roles`).
- Enforcing permissions on existing routes (guard logic in api-admin or admin frontend).
- Deactivating / soft-deleting admin users from the UI.
